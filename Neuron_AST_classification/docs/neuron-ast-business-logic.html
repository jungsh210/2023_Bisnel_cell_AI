// 01 Purpose
이 프로젝트가 실제로 하는 일

이 코드는 CNN 기반 이미지 분류 실험 파이프라인이다. 연구자가 준비한 이미지 데이터를 학습용, 검증용, 테스트용으로 나누어 읽고, 선택한 CNN 아키텍처를 학습시킨 뒤, 최종적으로 정확도와 손실값을 확인해 모델 성능을 판단하는 것이 핵심 목표다.

Problem: 이미지 분류 Model Family: CNN / torchvision backbone Runtime: PyTorch Output: Loss · Accuracy · Learning Curves

입력 관점

  • 입력은 이미지 폴더 구조를 가진 데이터셋이다.
  • 코드는 train, val, test 폴더를 기대한다.
  • 각 폴더 안에는 클래스별 하위 폴더가 있어야 한다.
  • 현재 기본 실험은 2-class 분류를 가정한다.

출력 관점

  • 학습 중 epoch별 train Loss / train Acc 출력
  • 검증 중 val Loss / val Acc 출력
  • 최종 test Loss / test Acc 출력
  • 마지막에 Accuracy / Loss 그래프 표시
핵심 해석: 이 프로젝트의 비즈니스 로직은 “데이터를 넣으면 분류 결과가 나온다”에서 끝나지 않는다. 실제 목적은 결과를 보고 현재 모델과 데이터 조합이 적절한지 판단하고, 다음 실험 조건을 정하는 데 있다.
경로 사용 원칙: 이 문서와 다른 md 문서에 등장하는 모든 경로는 예시다. 실제 실행 전에는 반드시 사용자 본인 컴퓨터의 경로 구조에 맞게 수정해야 한다.

// 03 Entry Point
실행 진입점과 실행 위치

실험은 프로젝트 루트에서 Experiments_Neuron.py를 실행하는 방식으로 시작한다. 즉, 이 파일이 전체 파이프라인의 오케스트레이터 역할을 한다.

실행 위치

프로젝트 루트 디렉터리에서 실행한다.

/path/to/Neuron_AST_classification_code

실행 명령 예시

python Experiments_Neuron.py

실행 전 반드시 des_path가 현재 데이터셋 위치와 맞는지 확인해야 한다.

왜 이 파일이 중요한가

실험 설정, 모델 선택, 데이터 로딩 호출, 학습 호출, 테스트 호출, 그래프 출력까지 전부 이 파일에서 연결된다.

가장 먼저 수정하는 값

des_path, model_name, num_classes, batch_size, num_epochs, feature_extract

초심자가 자주 놓치는 점

경로를 바꾸지 않고 실행하거나, 실제 클래스 수와 num_classes가 맞지 않아 오류가 나는 경우가 많다.

// 04 Flow Logic
Experiments_Neuron.py 기준 실제 호출 흐름

아래 흐름은 코드가 실제로 어떤 순서로 실행되는지 보여준다. 처음 받은 사람은 이 부분을 이해하면 전체 프로젝트 구조를 거의 다 잡은 셈이다.

Function Call Chain
main 시작버전 출력, device 결정
실험 설정des_path, model_name 등
initialize_model()TrainModels.py
get_data_loader()LoadData.py
train_model()TrainModels.py
test_model()TestModel.py
plt.show()결과 그래프 확인

단계 1. 실험 설정

  1. des_path로 데이터 루트를 지정한다.
  2. model_name으로 사용할 모델 구조를 고른다.
  3. num_classes, batch_size, num_epochs를 정한다.
  4. feature_extract로 전체 미세조정 여부를 정한다.

단계 2. 데이터 로딩

  1. LoadData.py가 이미지 폴더를 읽는다.
  2. 이미지를 224x224로 resize 한다.
  3. grayscale 처리와 normalization을 적용한다.
  4. 학습 코드가 사용할 DataLoader를 만든다.

단계 3. 학습과 검증

  1. train_model()가 epoch 반복을 시작한다.
  2. 각 epoch에서 train과 val을 순서대로 돈다.
  3. loss와 accuracy를 계산하고 출력한다.
  4. 가장 좋은 validation accuracy의 weight를 보관한다.

단계 4. 테스트와 시각화

  1. test_model()이 test split으로 최종 평가한다.
  2. test Loss, test Acc가 출력된다.
  3. accuracy/loss history를 그래프로 그린다.
  4. 사용자는 수치와 그래프를 함께 해석한다.

// 06 Edit Policy
수정해도 되는 파일과 건드리지 않는 파일

처음 받은 사람이 가장 많이 헷갈리는 부분은 “무엇을 바꿔야 하는지”와 “무엇을 건드리면 안 되는지”다. 이 구분이 없으면 문제를 해결하려다가 저장소를 더 복잡하게 만들 수 있다.

먼저 수정하는 파일

  • Experiments_Neuron.py
  • 수정 대상: des_path, 모델 설정, epoch, batch size
  • 이 파일만 잘 이해해도 첫 실행은 가능하다

조건부 수정 파일

  • LoadData.py
  • 데이터 transform을 바꾸고 싶을 때만 수정
  • TrainModels.py
  • 모델 구조나 학습 로직을 바꾸고 싶을 때만 수정

처음에는 수정하지 않는 파일

  • .git/
  • .history/
  • __pycache__/
  • .vscode/
  • LibPhySigProc/.git

왜 수정하면 안 되는가

  • 실험 로직이 아니라 도구/캐시/버전관리 정보이기 때문이다
  • 실행 결과에는 영향을 주지 않거나, 잘못 건드리면 오히려 문제를 만든다
// 07 Output Logic
결과가 무엇을 의미하는가

이 프로젝트의 로직은 “모델을 학습시켜 숫자를 뽑는다”가 아니라, 뽑힌 숫자와 그래프를 보고 현재 실험이 유효했는지 판단하는 데 있다.

콘솔 결과

  • train Loss: 훈련 데이터에서의 오차
  • train Acc: 훈련 데이터에서의 정확도
  • val Loss: 검증 데이터에서의 오차
  • val Acc: 검증 데이터에서의 정확도
  • test Loss / test Acc: 최종 평가 결과

그래프 결과

  • Accuracy 그래프: 학습이 진행될수록 성능이 좋아지는지 확인
  • Loss 그래프: 오차가 안정적으로 줄어드는지 확인
  • train과 val 차이가 너무 벌어지면 과적합 가능성 검토
현재 구조의 특징: 결과는 별도 리포트 파일보다 콘솔 로그와 matplotlib 창에 주로 나타난다. 따라서 실행 후 사용자가 직접 실험 조건과 결과를 기록해 두는 것이 중요하다.
// 08 Validation Checklist
정상 실행 체크리스트

처음 받은 사람이 “내가 제대로 돌리고 있는가?”를 빠르게 판단할 수 있도록 정상 신호와 이상 신호를 분리한다.

시작 직후 정상 신호
  • PyTorch / Torchvision 버전이 출력된다
  • 모델 구조가 출력된다
  • Initializing Datasets and Dataloaders...가 보인다
데이터 로딩 정상 신호
  • train/test 이미지 시각화 창이 뜬다
  • 경로 오류가 나지 않는다
  • 클래스 폴더를 정상적으로 읽는다
학습 정상 신호
  • Epoch 0/... 로그가 진행된다
  • train Loss, train Acc가 반복 출력된다
  • val Loss, val Acc가 함께 출력된다
평가 정상 신호
  • 마지막에 test Loss와 test Acc가 출력된다
  • Accuracy / Loss 그래프가 뜬다
가장 흔한 이상 신호
  • des_path 경로 오류
  • train/val/test 폴더 구조 불일치
  • 클래스 수와 num_classes 불일치
  • 그래프가 뜨기 전에 실행이 중단됨
이상이 있을 때 먼저 볼 곳
  • Experiments_Neuron.py의 des_path
  • 데이터셋의 폴더 구조
  • LoadData.py transform 및 입력 형태
// 09 Structural Risks
현재 비즈니스 로직의 구조적 한계

코드가 실행된다고 해서 운영이나 후속 연구 준비가 끝난 것은 아니다. 현재 프로젝트는 실험 파이프라인으로서는 동작하지만, 재현성과 인수인계 관점에서는 보완이 필요한 구조를 갖고 있다.

1. 경로 하드코딩

des_path가 코드 안에 직접 들어 있어 환경이 바뀌면 바로 실행이 깨진다.

2. 설정 분리 부족

실험 설정이 별도 config가 아니라 메인 파일 내부 값에 묶여 있다.

3. 결과 저장 부족

실험 결과가 주로 콘솔과 그래프로만 제공되어 비교 기록이 어렵다.

4. 초심자 진입장벽

메인 구조를 모르면 어떤 파일부터 봐야 하는지 판단하기 어렵다.

5. 보조 파일 혼재

전처리, 기록 파일, 서브모듈이 함께 있어 목적이 다른 파일들을 구분해야 한다.

6. 후속 과제

문서화 이후에는 config화, 결과 저장, 경로 분리, 실험 로그 체계화가 필요하다.
